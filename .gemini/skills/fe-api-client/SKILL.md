# API Client (consuming MEDA Java services)

## The MEDA convention (critical)
Java internal RPC: **POST + JSON**, wrapped in `APIRequest<T>` / `APIResponse<T>`. Business errors
come back as **HTTP 200 with `status: "ERROR"`** + `errorCode`/`errorMessage`. Unknown errors are
HTTP 500. So the client MUST check `status`, not just the HTTP status. Response VO fields are strings.

## Base client
```typescript
// lib/api/client.ts
interface APIResponse<T> { status: 'OK' | 'ERROR'; errorCode: string | null; errorMessage: string | null; data: T; }

export class MedaApiError extends Error {
  constructor(public code: string | null, message: string | null) { super(message ?? 'API error'); }
}

export async function post<T>(path: string, body: unknown): Promise<T> {
  const res = await fetch(`${BASE_URL}${path}`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json', ...authHeader() },
    body: JSON.stringify({ traceId: genTraceId(), body }),  // APIRequest envelope
  });
  if (res.status === 500) throw new MedaApiError('SYSTEM_ERROR', 'Service unavailable');
  const json = (await res.json()) as APIResponse<T>;
  if (json.status === 'ERROR') throw new MedaApiError(json.errorCode, json.errorMessage);
  return json.data;
}
```

## Per-service functions
One file per service under `lib/api/` (or per feature under `features/x/api/`). Typed in/out
mirroring the Java contract. Generated by `/meda-fe-endpoint` from OpenAPI/Postman/Java repo.

## Getting the contract
Prefer an OpenAPI/Swagger spec or Postman collection as the source of truth (the team maintains a
collection). Generate types from it. Fall back to reading the Java VOs, or asking the dev.

## Rules
- ALWAYS handle status:"ERROR" — HTTP 200 does not mean success in MEDA.
- Type request and response; no `any`.
- Auth token attached by the client (`fe-auth`), never hardcoded.
- traceId sent in the request envelope for end-to-end tracing with the backend.
